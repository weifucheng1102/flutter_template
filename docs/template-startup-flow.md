# 启动链路说明

这个模版不是直接打开首页，而是通过启动分流页决定用户应该进入哪里。

## 流程图

```text
main.dart
  -> GetMaterialApp(initialRoute: /)
    -> BootstrapPage
      -> 未同意协议 -> AgreementNotice
      -> 已同意协议且未登录 -> LoginPage
      -> 已登录 -> LaunchPage
```

## 各节点职责

### `main.dart`

负责：

- 初始化 Flutter 绑定
- 初始化 `GetStorage`
- 注册主题控制器
- 注册 Bruno 主题
- 启动 `GetMaterialApp`

不负责：

- 判断登录态
- 判断协议状态
- 执行业务请求

### `BootstrapPage`

负责：

- 读取协议确认状态
- 读取登录态
- 在必要时触发 iOS 网络预热
- 决定进入协议页、登录页还是主壳页面

这是启动逻辑的唯一分流入口。

### iOS 首次网络访问说明

这里处理的是 iOS 首次数据网络访问时的系统授权场景，不是 `permission_handler` 可直接申请的那类权限。

模板当前做法：

- 已同意协议后，启动阶段会尝试做一次轻量网络预热
- 如果首个业务请求正好因为这个授权时机失败，请求层会在预热完成后自动重试一次

这样做的目的，是避免“用户已经同意联网，但第一次真实业务请求已经失败且没有自动补发”的问题。

### `AgreementNotice`

负责：

- 显示协议确认弹窗
- 跳转用户协议/隐私政策文档页
- 在用户确认后写入协议状态

建议新项目第一时间替换协议文案。

### `LoginPage`

当前是模版占位页。

负责：

- 演示登录页路由
- 演示登录态写入
- 演示登录后跳转主壳页面

建议新项目第一时间替换成真实登录流程。

### `LaunchPage`

负责承接真正的业务壳页面。

当前 `LaunchPage -> NavigatePage`，而 `NavigatePage` 只是一个基础底部导航骨架。

## 为什么这样设计

因为模板项目需要稳定地支持这些场景：

- 首次安装必须先同意协议
- 未登录用户不能直接进业务首页
- 登录后可以回到统一主壳

如果把这些判断写散在多个页面里，后续项目会很快失控。

## 新项目改造建议

新项目接入时，优先按这个顺序改：

1. 改 `AppConfig`
2. 改 `AgreementNotice`
3. 改 `LoginPage`
4. 改 `NavigatePage`
5. 追加业务路由

## 不建议的改法

- 不要把首页直接塞回 `main.dart`
- 不要跳过 `BootstrapPage`
- 不要在多个页面各自判断 token
- 不要在登录成功后直接写死跳转具体业务页，优先回主壳
